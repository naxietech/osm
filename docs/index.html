Getting Started

Scaffolding the OSES Project

A complete, step-by-step guide to get the OSES monorepo running on your local machine — from cloning the repository to seeing the API and web app live in the browser.

Prerequisites

Install all tools listed below before you proceed. Every version constraint is intentional.

🟩
Node.js
>= 20 LTS
The JavaScript runtime. v20 is required for modern ESM and crypto APIs.
📦
pnpm
>= 9
Fast, disk-efficient package manager. Powers the monorepo workspaces.
🔧
Git
>= 2.39
Version control. Husky hooks require a git repository.
🐍
Python
3.11 (future)
Only needed for the OpenCV worker — not required for this scaffold.
💡
Install pnpm globally: run npm install -g pnpm after installing Node. Then verify: pnpm --version should print 9.x.x.

Installation Steps

1

Clone the Repository

Clone the project and move into the monorepo root.

# Clone (replace URL with your actual remote)
git clone <repository-url> oses-project
cd oses-project/oses

The monorepo root is the oses/ folder — this is where turbo.json, pnpm-workspace.yaml, and the root package.json live. Always run commands from here, not from inside individual apps.

2

Install All Dependencies

A single pnpm install at the root installs dependencies for every package and app in the monorepo simultaneously. 

pnpm install

What this does internally:

  • Reads pnpm-workspace.yaml to discover all workspace members: apps/* and packages/*
  • Downloads all external dependencies (React, NestJS, Vitest…) into a shared global store
  • Creates symlinks so that @oses/types, @oses/eslint-config, and other local packages resolve directly to the source folders — no build step needed
  • Generates a pnpm-lock.yaml file that pins exact versions for reproducible installs across all machines
ℹ️

You will see output for each workspace member. This is normal. The entire install should complete in under 30 seconds on a warm pnpm store (subsequent installs are faster because packages are cached globally).

3

Activate Git Hooks (Husky)

This registers the .husky/pre-commit script with Git. Without this step, the pre-commit hook file exists but Git will never call it. 

pnpm husky install

After this, every git commit will automatically run lint-staged, which applies ESLint + Prettier to all staged .ts and .tsx files before the commit goes through. This prevents unformatted or type-unsafe code from ever entering the repository.

⚠️

This step only needs to be run once per machine after you clone the repo. It does not need to be repeated on updates.

4

Create Environment Files

The .env.example files are committed to Git as documentation. The actual .env files are gitignored — they hold secrets. Copy the examples to create your local environment files. 

# API environment file
cp apps/api/.env.example apps/api/.env

# Web environment file
cp apps/web/.env.example apps/web/.env

apps/api/.env — contents

PORT=3001
NODE_ENV=development
JWT_SECRET=change-this-to-a-secure-random-string-minimum-32-characters
JWT_EXPIRES_IN=15m
CORS_ORIGIN=http://localhost:5173

apps/web/.env — contents

VITE_API_BASE_URL=http://localhost:3001/api/v1
VITE_APP_NAME=OSES
VITE_ENV=development
5

Set a Secure JWT Secret

Open apps/api/.env and replace the placeholder JWT_SECRET with a real random string of at least 32 characters. This is the key used to sign and verify all authentication tokens in the system. 

# Generate a secure secret — run this and paste the output into .env
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
🔒

Never commit .env to Git. The .gitignore already excludes it, but double-check before pushing. A leaked JWT secret allows anyone to forge authentication tokens.

6

Start the Development Servers

Run this single command from the monorepo root. Turborepo will start both the NestJS API and the Vite web app in parallel, streaming their output into one terminal. 

turbo dev

You will see interleaved output from both processes:

@oses/api:dev:  [Nest] Starting Nest application...
@oses/web:dev:  VITE v6.0 ready in 312ms
@oses/api:dev:  OSES API running on http://localhost:3001/api/v1
@oses/web:dev:    ➜  Local:   http://localhost:5173/

Both servers have hot-reload — saving any file triggers an instant update without restarting.

7

Verify Everything is Running

Open these URLs in your browser to confirm both services are healthy:

5173
Web App (Vite)
http://localhost:5173
React frontend. Should show the OSES login page.
3001
API (NestJS)
http://localhost:3001/api/v1
NestJS backend. Returns a 404 JSON at root — that is expected.
3001
Swagger Docs
http://localhost:3001/api/docs
Interactive API explorer. Try the login endpoint here.
✅

Quick smoke test: Open Swagger at http://localhost:3001/api/docs, expand POST /api/v1/auth/login, send body {"email":"admin@oses.pk","password":"password123"} — you should receive a 200 response with an accessToken.

Architecture & Decisions

Tech Stack Selection

Every tool and library in OSES was chosen deliberately. This document explains the "why" behind each decision — not just what it is, but why it was the right choice for a national-scale examination platform serving 1,000,000+ students.

🏗️ Monorepo Tooling

pnpm Tooling
v9+
Standard npm creates a separate node_modules copy per project, wasting gigabytes. pnpm stores every package once in a global content-addressable store on your machine and uses hard links in each project. This makes install 2–4× faster and the entire monorepo fits in a fraction of the disk space.

pnpm also has built-in workspace support that lets local packages like @oses/types be symlinked directly into both apps/api and apps/web without publishing to npm. Changes to shared types are instantly visible across all apps.
↳ Chosen over npm/yarn for speed, disk efficiency, and workspace quality
Turborepo Tooling
v2+
Without Turborepo you would manually decide: "build types before api before web." Turbo reads the dependency graph automatically from each package's package.json and executes tasks in the correct order, in parallel where possible.

More importantly, Turbo caches task outputs. If you run turbo build and nothing changed in packages/types, it skips rebuilding it — restoring the dist/ from cache in milliseconds. On CI this can cut build times from 5 minutes to under 30 seconds.
↳ Chosen over Nx/Lerna for simplicity, zero-config caching, and Vercel backing

🔷 Core Language

TypeScript Shared
v5, strict mode
OSES handles sensitive examination data and student PII. A runtime bug that exposes a student's CNIC to an evaluator is a legal and ethical failure. TypeScript's strict mode catches entire categories of mistakes at compile time — null dereferences, missing fields in an API response, wrong types passed across the frontend/backend boundary.

The packages/types package lets the same type definitions (e.g., School, SafeUser) be shared between the NestJS API and the React frontend. The compiler enforces the contract — if an API response shape changes, the frontend fails to compile immediately.
↳ Non-negotiable for a safety-critical national exam platform

⚛️ Frontend

React 18 Frontend
v18.3
OSES has complex interactive UIs: on-screen answer marking with real-time state, data tables with thousands of rows, multi-step exam workflows. React's component model makes it straightforward to build and maintain these views as composable, testable units.

React 18's Concurrent Features (used via React.Suspense for lazy-loaded routes) keep the UI responsive even when loading large marking screens. The ecosystem — React Testing Library, React Router, React Hook Form — is mature and stable.
↳ Chosen for ecosystem maturity, team familiarity, and Concurrent Mode
Vite Frontend
v6
Vite uses native ES modules in the browser during development — it never bundles the entire app on startup. When you save a file, only that module is re-sent to the browser. Hot Module Replacement (HMR) feels instant even in large projects.

In production, Vite uses Rollup to produce optimised, tree-shaken bundles. The developer experience gap between Vite and a Webpack-based setup (like CRA) is enormous on a project of this scale.
↳ Chosen over Create React App / Webpack for HMR speed and modern ESM defaults
Tailwind CSS Frontend
v4
OSES has a custom brand (navy #1B3A6B, teal #0E7490, gold #B45309) and needs pixel-perfect, consistent spacing across many screens. Writing custom CSS for each component is slow and leads to inconsistency. Using a UI library like Material UI locks you into someone else's design language and adds bundle weight.

Tailwind gives us utility-first classes that compose directly in JSX, eliminating context-switching between CSS files. The production build purges unused classes — the final CSS is typically under 10 KB.
↳ Chosen over MUI/Chakra for design freedom and zero runtime overhead
Atomic Design System Frontend
Custom
OSES will eventually have 20+ screens built by multiple developers. Without a design system, every developer recreates buttons, inputs, and tables differently — the UI becomes inconsistent and unmaintainable.

Atomic Design defines a clear hierarchy: Atoms → Molecules → Organisms → Templates → Pages. A Button atom is built once, tested once, and used everywhere consistently. When the brand color changes, you update one file. The strict import rules (atoms cannot import from molecules) enforce separation of concerns automatically.
↳ Chosen for consistency across many screens and many developers
React Router v6 Frontend
v6.28
OSES has role-based routing — an admin sees different pages than an evaluator. React Router v6's Outlet and nested route model makes it trivial to build ProtectedRoute and RoleRoute wrappers that sit between the URL and the page component. All routes are code-split with React.lazy — evaluators never download the admin bundle.
↳ Industry standard; Outlet model ideal for role-based access wrappers

⚙️ Backend

NestJS Backend
v10 — Express adapter
NestJS provides an opinionated, Angular-inspired structure (modules, controllers, services, guards) that makes large APIs consistent and easy to navigate for any developer on the team. Without it, plain Express projects grow into unmaintainable spaghetti as the API grows.

It has first-class support for Swagger/OpenAPI, JWT Passport strategies, and dependency injection — all things OSES needs. We use the Express adapter (not Fastify) for maximum middleware compatibility and a lower learning curve.
↳ Chosen over plain Express for structure, DI, and team scalability
Zod Backend
v3
Every piece of data that enters the API (from browsers, mobile apps, or Postman) is untrusted input. Zod validates and parses it at the boundary. Unlike class-validator, Zod schemas produce a TypeScript type via z.infer<> — you never manually write both a class and a type.

Zod transforms also coerce and sanitise data (e.g., trim whitespace, lowercase emails) in the same pass as validation. The ZodValidationPipe in shared/pipes wires this into NestJS seamlessly.
↳ Chosen over class-validator for type inference and runtime+compile-time safety
Passport + JWT Backend
@nestjs/passport
OSES has four user roles (Admin, Controller, Evaluator, School Staff) each with different screen access and data visibility rules. JWTs carry the role in the token payload — the RolesGuard reads it without hitting the database on every request.

The Passport jwt strategy integrates cleanly with NestJS guards. JwtAuthGuard protects endpoints; RolesGuard restricts by role; the @CurrentUser() decorator extracts the payload cleanly.
↳ Standard NestJS auth pattern; JWT avoids session state on the server
@nestjs/swagger Backend
v7
OSES is a multi-team project. Frontend developers, QA engineers, and future mobile developers all need to understand and test the API without reading source code. Swagger generates a live, interactive API explorer at /api/docs automatically from NestJS decorators. It documents every endpoint, request body, and response shape.
↳ Eliminates need for separate API documentation; always in sync with code

🧪 Testing

Vitest Testing
v2 — Frontend
Vitest is Vite-native — it runs tests in the same Vite pipeline that the dev server uses. This means it respects your path aliases (@/), TypeScript config, and environment variables with zero extra configuration. Tests run in parallel and complete in under a second for the component test suite.
↳ Chosen over Jest for frontend because it shares the Vite build config directly
React Testing Library Testing
v16
RTL tests components the way users interact with them — by clicking buttons, reading text, and querying by accessible role — not by inspecting internal state or component instance methods. This means tests remain valid even when you refactor component internals completely.
↳ Enforces accessible, behaviour-driven tests rather than implementation tests
Jest Testing
v29 — Backend
NestJS's testing utilities (@nestjs/testing) are built around Jest. The Test.createTestingModule() API compiles the NestJS DI container in a test environment, allowing true integration tests of controllers + services + guards without starting a real HTTP server.
↳ NestJS's native test runner; deep integration with @nestjs/testing

✅ Code Quality

ESLint Quality
v8
ESLint enforces rules that the TypeScript compiler alone cannot catch: no any, no floating promises, no unused variables, explicit return types on exported functions. These rules prevent entire classes of bugs common in large codebases. The shared @oses/eslint-config ensures identical rules across the API and web app without duplication.
↳ Catches logic errors and style drift that TypeScript's type system doesn't cover
Prettier Quality
v3
Code formatting debates (tabs vs spaces, semicolons, quote style) waste engineering time and pollute git diffs. Prettier ends the debate — it reformats every file to a single canonical style on every save and commit. Diffs only show logic changes, not formatting noise.
↳ Eliminates formatting disagreements and keeps diffs clean
Husky + lint-staged Quality
Husky v9
Rules are only useful if they are enforced. Husky runs lint-staged as a Git pre-commit hook — before any commit is allowed to complete, ESLint and Prettier are run on staged files. Developers cannot accidentally commit broken or unformatted code even if they forgot to run the linter manually.

lint-staged ensures only changed files are linted — not the entire codebase — so the hook completes in under 2 seconds.
↳ Makes code quality rules automatic and unavoidable at commit time

Standards & Conventions

Naming Conventions

Consistent naming removes cognitive load. When every developer follows the same rules, you can read any file in the codebase and instantly know what role it plays. These conventions are enforced by ESLint where possible — not just documented.

ℹ️

In the tables below, ✓ Correct shows the expected pattern and ✗ Wrong shows what to avoid.

📄 Files

Rule: All files use kebab-case.

Why: Kebab-case is the only file naming convention that works identically on Linux (case-sensitive), macOS (case-insensitive by default), and Windows. PascalCase filenames like SchoolList.tsx can cause import errors when the same code runs on a Linux CI server — schoolList.tsx and SchoolList.tsx would both work locally on Mac but break on Linux. Kebab-case eliminates this entire class of cross-platform bugs.

React componentschool-list.page.tsxSchoolList.tsx, schoolList.tsx
NestJS serviceschools.service.tsSchoolsService.ts
NestJS controllerschools.controller.tsschoolsController.ts
NestJS moduleschools.module.tsschools_module.ts
DTO filecreate-school.dto.tsCreateSchool.dto.ts
Test filebutton.test.tsxButton.test.tsx, buttonTest.tsx

The file name suffix also communicates the file's role at a glance. A developer seeing schools.service.ts instantly knows it is a NestJS service without opening it.

⚛️ React Components

Rule: Component names (the function or class) use PascalCase.

Why: React requires PascalCase for components — JSX treats lowercase names as native HTML elements. <button> is an HTML button; <Button> is your React component. The file is kebab-case (button.tsx); the export is PascalCase (Button). This two-layer convention is intentional.

💡

Every component must have both a named export and a default export. Named exports are used in barrel files and within the same package. Default exports are used by React.lazy() for code-splitting.

// button.tsx — both exports required
export function Button(...) { ... }   // named
export default Button;              // default

🔒 Constants

Rule: Module-level constants that are truly fixed values use UPPER_SNAKE_CASE.

Why: UPPER_SNAKE_CASE is a signal to every reader that this value is fixed at compile time and should never be mutated. It distinguishes configuration values from runtime variables.

⚠️

Only use UPPER_SNAKE_CASE for module-level constants, not for local variables inside functions — even if they don't change. A local const result = ... inside a function stays camelCase.

🔷 TypeScript Interfaces & Types

Rule: PascalCase. No I-prefix (no ISchool).

Why: The I-prefix is a C# convention that TypeScript's own style guide explicitly discourages. TypeScript's type system already distinguishes interfaces from values — you never need a prefix to tell them apart. School is cleaner and reads more naturally than ISchool.

Generic paramT, TDatat, tData
Props interfaceButtonProps, FormFieldPropsIButtonProps, buttonProps

🏷️ TypeScript Enums

Rule: Enum name is PascalCase. Enum values are UPPER_SNAKE_CASE.

// ✓ Correct
export enum UserRole {
  ADMIN        = 'ADMIN',
  CONTROLLER   = 'CONTROLLER',
  EVALUATOR    = 'EVALUATOR',
  SCHOOL_STAFF = 'SCHOOL_STAFF',
}

// ✓ Correct
export enum OnboardingStatus {
  PENDING     = 'pending',
  IN_PROGRESS = 'in_progress',
  COMPLETE    = 'complete',
  SUSPENDED   = 'suspended',
}

// ✗ Wrong — never do this
enum userRole { admin = 'admin' }        // lowercase enum name
enum UserRole { Admin = 'admin' }       // PascalCase values

📦 Barrel Files (index.ts)

Rule: Every folder must have an index.ts that re-exports everything from the files in that folder. Never import directly from deep paths.

✓ Correct — use barrel

// Importing from the barrel
import { Button } from
  '@/design-system/atoms/button';

✗ Wrong — deep path import

// Never import the implementation directly
import { Button } from
  '@/design-system/atoms/button/button';

Barrel files also mean you can move or rename implementation files without breaking every import across the codebase — you only update the barrel.

Why This Structure?

📦
packages/ vs apps/

packages/ contains code that other packages consume — shared types, shared lint rules, shared TS config. It has no HTTP server, no UI, no entry point. apps/ contains runnable applications. This distinction means packages/types can never accidentally import from apps/api — that would invert the dependency graph. TypeScript will catch it, and Turborepo's task pipeline enforces the build order.

🔀
shared/ inside the API

src/shared/ holds infrastructure that is cross-cutting — it is used by every module but belongs to no single domain. Filters, interceptors, pipes, and decorators live here. Keeping them outside any feature module prevents circular imports and makes them trivial to find: "where's the error shaping code?" → always shared/filters/.

⚛️
Atomic Design layers — strict import rules

The design-system is layered by complexity and dependency direction. The rule is simple: a layer may only import from layers below it.

🆕 Creating a New Module — Complete Example

The schools module is the reference pattern. When a developer needs to add a new domain — for example, Exam Cycles — they follow these four steps. Nothing is invented; every file mirrors the schools pattern exactly.

📋

Example: Add an "Exam Cycles" module so admins can create and manage exam cycles (Jan 2026, Jun 2026, etc.).

1

Add shared types to packages/types/

Types live in packages/types/src/ so both the API and web app share the same contract. Create exam-cycle.types.ts:

// packages/types/src/exam-cycle.types.ts
export enum ExamCycleStatus {
  DRAFT  = 'draft',
  ACTIVE = 'active',
  CLOSED = 'closed',
}

export interface ExamCycle {
  id:        string;
  cycleCode: string;
  cycleName: string;
  startDate: string;
  endDate:   string;
  status:    ExamCycleStatus;
  createdAt: string;
  updatedAt: string;
}

export interface CreateExamCycleDto {
  cycleCode: string;
  cycleName: string;
  startDate: string;
  endDate:   string;
}

Then export it from the barrel packages/types/src/index.ts:

export * from './exam-cycle.types';  // add this line

DTO — dto/create-exam-cycle.dto.ts

import { z } from 'zod';

export const CreateExamCycleDtoSchema = z.object({
  cycleCode: z.string().min(3).max(20),
  cycleName: z.string().min(5).max(100),
  startDate: z.string().datetime(),
  endDate:   z.string().datetime(),
});

export type CreateExamCycleDto = z.infer<typeof CreateExamCycleDtoSchema>;

Service — exam-cycles.service.ts (stub pattern, same as schools)

import { Injectable, NotFoundException } from '@nestjs/common';

import { ExamCycleStatus } from '@oses/types';

@Injectable()
export class ExamCyclesService {
  async findAll(): Promise<ExamCycle[]> {
    return []; // TODO: replace with DB query
